## Überblick

Die meisten KI-Coding-Tools arbeiten in einer einzelnen Konversation: Man beschreibt, was man will, das Modell schreibt Code, und man hofft auf das Beste. Bei allem, was nicht trivial ist, bricht das schnell zusammen. Das Modell halluziniert APIs, überspringt Tests, macht destruktive Änderungen am Repo und produziert Code, der kompiliert, aber in Produktion scheitert. Keine Struktur, keine Verifikation, nichts hält es davon ab, `rm -rf` auf das Arbeitsverzeichnis loszulassen.

devline ist ein [Claude Code](https://docs.anthropic.com/en/docs/claude-code)-Plugin, das diesen Ansatz durch eine Multi-Agenten-Pipeline ersetzt. Statt einer einzelnen Konversation für alles übernehmen spezialisierte Agenten jede Phase, vom Brainstorming über die Implementierung bis zum Review. Sicherheits-Hooks erzwingen Branch Protection, blockieren destruktive Befehle und fangen Credential Leaks ab, bevor sie auf die Platte geschrieben werden — so kann man die Agenten laufen lassen, ohne ihnen ständig über die Schulter schauen zu müssen.

Ich habe devline gebaut, weil ich klar definierte Arbeit an KI delegieren wollte, ohne ständig prüfen zu müssen, ob etwas kaputtgegangen ist. Kein Vibe-Coding, sondern Spec-getriebene Planung, striktes TDD und eine Review-Schleife, die Fehler abfängt, bevor sie den Branch erreichen.

## Wie die Pipeline funktioniert

Die ersten beiden Phasen sind interaktiv — man bleibt eingebunden. Alles danach läuft autonom.

**Brainstorm** — Man beschreibt ein Feature in natürlicher Sprache. Der Brainstormer stellt ein paar gezielte Fragen zu Scope und Grenzfällen und liefert dann eine kurze Spezifikation.

**Plan** — Ein Planner entwirft die Architektur TDD-first, schlägt Bibliotheksdokumentation in Echtzeit über Context7 nach, teilt die Arbeit in parallele Pakete mit Dateiisolierung auf und schreibt den Plan auf die Festplatte. Er hinterfragt seinen eigenen Ansatz und schlägt Alternativen vor, bevor man ihn freigibt.

**Implement** — Ein Agent pro Arbeitspaket, parallel laufend. Jeder folgt striktem TDD: fehlschlagende Tests schreiben, implementieren bis grün, refactoren. Kein Produktionscode existiert ohne Test.

**Review** — Jedes Arbeitspaket wird auf Korrektheit, Sicherheit und Performance geprüft. Fällt das Review durch, bekommt der Implementierer einen zweiten Versuch. Scheitert auch der, übernimmt ein Debugger-Agent die Ursachenanalyse, bevor an den Entwickler eskaliert wird.

![Implementer und Reviewer laufen parallel am Schachprojekt](/images/projects/devline/chess-development-example.png)

**Docs** — Ein Docs-Keeper-Agent erkennt, welche Dateien sich geändert haben, und aktualisiert sie.

**Deep Review** — Letztes Gate vor dem Merge: Security-Audit, Credential-Scanning, Codequalität, technische Schulden und Prüfung, ob jedes Akzeptanzkriterium aus dem Plan erfüllt wurde. Fehler gehen zurück an den Implementierer; architektonische Probleme werden an den Planner eskaliert.

**Du entscheidest** — Freigeben, Änderungen anfordern, die zurück in die Pipeline gehen, oder devline automatisch committen und mergen lassen.

## Agenten-Architektur

devline steuert 10 spezialisierte Agenten, jeweils mit einem bestimmten Modell-Tier (Opus für reasoning-intensive Aufgaben, Sonnet für Durchsatz) und vorgeladenen Domain-Skills:

| Agent | Modell | Rolle |
|---|---|---|
| Planner | Opus | Architekturentwurf, TDD-Planung, interaktives Q&A |
| Implementer | Sonnet | Parallele TDD-Implementierung pro Arbeitspaket |
| DevOps | Sonnet | Infrastruktur, CI/CD, Docker, Build-Systeme |
| Reviewer | Sonnet | Korrektheit, Sicherheit, Performance-Review |
| Debugger | Opus | Systematische Ursachenanalyse nach wissenschaftlicher Methode |
| Deep Review | Opus | Finales Merge-Gate-Audit |
| Frontend Reviewer | Sonnet | Automatisch ausgelöste Barrierefreiheits- und UI-Qualitätsprüfungen |
| Docs-keeper | Vererbt | Dokumentationspflege |
| Dependency Patcher | Sonnet | CVE-Patching über Repositories hinweg |
| Dependency Migrator | Opus | Breaking-Change-Migrationen mit Code-Refactoring |

Agenten werden ausschließlich über den Pipeline-Orchestrator oder einen expliziten Befehl aufgerufen. Fachwissen kommt aus komponierbaren Skill-Modulen — TDD-Methodik, Frontend-Patterns, Cloud-Infrastruktur, Debugging-Heuristiken — sodass jeder Agent nur den Kontext mitbekommt, den er tatsächlich braucht.

## Sicherheits-Hooks

devline ist für Claude Codes Bypass-Permissions-Modus konzipiert, in dem Agenten mit voller Autonomie und ohne Bestätigungsdialoge arbeiten. Die Hooks sind das Einzige zwischen den Agenten und der Codebasis:

- Schreibzugriffe auf geschützte Branches (main, master, develop, release, production, staging) sind blockiert. Agenten müssen zuerst einen Feature-Branch erstellen.
- Destruktive Befehle — `rm -rf` außerhalb des Arbeitsverzeichnisses, Force-Pushes, Hard-Resets, Datenbank-Drops, Paket-Publishing — werden hart blockiert.
- Hartcodierte API-Keys, Private Keys, AWS-Credentials, GitHub-Tokens und Secrets in `.env`-Dateien werden abgefangen, bevor sie auf die Platte geschrieben werden.
- Arbeitsdateien in `.devline/` können nicht gestaged oder committet werden.
- Conventional-Commit-Format ist standardmäßig erzwungen, pro Projekt konfigurierbar.
- HTTP-Mutationen zu Nicht-Localhost, SSH-Zugriff und Service-Control-Befehle brauchen explizite Bestätigung oder werden komplett blockiert.

## Praxiseinsatz

Ich habe devline ausgiebig an meinem [Schachprojekt](/de/projects/chess-engine) eingesetzt, einer Multi-Modul-Java-Plattform, die ich ursprünglich im dritten Semester gebaut habe. Damit habe ich die gesamte Oberfläche von Java Swing zu JavaFX migriert, ein animiertes cinematisches Hauptmenü gebaut und den Server von einem Proof-of-Concept zu einem produktionsfähigen System mit Authentifizierung, Spielstandspeicherung und gleichzeitiger Session-Verwaltung umgebaut. Die Pipeline hat Planung, paralleles TDD und Review-Zyklen für all das übernommen.

devline ist auch mein Test, ob strukturierte KI-Workflows im professionellen Teamkontext funktionieren können. Ich habe es mit dem Ziel gebaut, diesen Ansatz in mein Team bei [OTTO](https://www.otto.de) zu bringen.

## Installation

Über das Claude-Code-Plugin-System installierbar:

```
claude plugin add devline
```

Oder während der Entwicklung aus einem lokalen Verzeichnis:

```
claude plugin add /path/to/devline
```

`/devline` in einer beliebigen Konversation eingeben, um zu prüfen, ob alles läuft.
