## Overview

Most AI coding tools work in a single conversation: you describe what you want, the model writes code, and you hope for the best. On anything non-trivial, this breaks down fast. The model hallucinates APIs, skips tests, makes destructive changes to your repo, and produces code that compiles but fails in production. No structure, no verification, nothing stopping it from running `rm -rf` on your working directory.

devline is a [Claude Code](https://docs.anthropic.com/en/docs/claude-code) plugin that replaces this with a multi-agent pipeline. Every feature goes through spec engineering, TDD planning, parallel implementation, automated review, and documentation. Each stage runs on a specialized agent with the appropriate model, tools, and domain knowledge. Security hooks enforce branch protection, block destructive commands, and catch credential leaks before they hit disk, so you can let the agents run without babysitting them.

I built this because I wanted to delegate well-defined but time-consuming work to AI without constantly checking whether it broke something. Not vibe-coding. Spec-driven planning, strict TDD, and a review loop that catches problems before they reach your branch.

## What it does

The pipeline is interactive where it matters (brainstorming, planning, approval gates) and autonomous where it doesn't (implementation, review, documentation). You describe a rough idea, refine it through a short dialogue, approve a plan, and walk away. The agents do the rest.

**Brainstorm** — You describe a feature in plain language. The brainstormer asks 1-3 targeted questions to pin down scope, edge cases, and constraints, then produces a short spec.

**Plan** — A TDD-first planner designs the architecture, looks up libraries via real-time documentation (Context7 MCP), breaks work into parallel packages with file-level isolation, and writes the full plan to disk. It questions its own approach and suggests alternatives. You review and approve before anything gets implemented.

**Implement** — Multiple agents launch in parallel, one per work package. Each follows strict TDD: write failing tests first, implement until green, refactor. Every line of production code has a test behind it before it exists.

**Review** — Each work package gets reviewed for correctness, security, performance, and code quality. If a review fails, the implementer gets another shot. If that fails too, a debugger agent does systematic root cause analysis. Only then does it escalate to you.

**Documentation** — A docs-keeper agent figures out which files need updating based on what changed, and updates them.

**Deep Review** — A final gate before merge: security audit, credential scanning, code quality, technical debt, convention checks, and verification that every criterion from the plan was actually met. If something fails, it sends the work back to the implementer. If the problem is architectural, it escalates to the planner.

**You decide** — Once everything passes, you get the final call. Approve and exit, request changes that go back into the pipeline, or let devline commit and merge automatically.

## Agent architecture

devline runs 10 specialized agents. Each has a specific model tier (Opus for reasoning-heavy work, Sonnet for throughput), a permission level, and injected domain skills:

| Agent | Model | Role |
|---|---|---|
| Planner | Opus | Architecture design, TDD planning, interactive Q&A |
| Implementer | Sonnet | Parallel TDD implementation per work package |
| DevOps | Sonnet | Infrastructure, CI/CD, Docker, build systems |
| Reviewer | Sonnet | Correctness, security, performance review |
| Debugger | Opus | Scientific method root cause analysis |
| Deep Review | Opus | Final merge gate audit |
| Frontend Reviewer | Sonnet | Auto-triggered accessibility and UI quality checks |
| Docs-keeper | Inherited | Documentation maintenance |
| Dependency Patcher | Sonnet | CVE patching across repositories |
| Dependency Migrator | Opus | Breaking-change migrations with code refactoring |

Nothing fires randomly. Every agent invocation goes through the pipeline orchestrator or an explicit command. Domain knowledge comes from composable skill modules (TDD methodology, frontend patterns, cloud infrastructure, debugging heuristics), so each agent knows its domain well without carrying context it doesn't need.

## Security layer

devline is built for Claude Code's bypass permissions mode, where agents run with full autonomy and no approval prompts. That means the security hooks are the only thing between autonomous agents and your codebase:

- **Branch protection** — Writes are blocked on protected branches (main, master, develop, release, production, staging). Agents must create a feature branch first.
- **Destructive command blocking** — `rm -rf` outside the working directory, force pushes, hard resets, disk-level operations, database drops, and package publishing are all hard-blocked.
- **Credential scanning** — Hardcoded API keys, private keys, AWS credentials, GitHub tokens, JWTs, and secrets in `.env` files are blocked from being written to disk.
- **Pipeline isolation** — Working files in `.devline/` can't be staged or committed.
- **Commit validation** — Conventional commit format enforced by default, configurable per project.
- **External mutation guards** — HTTP mutations to non-localhost, SSH access, and service control commands need explicit confirmation or are blocked outright.

## Real-world use

I've used devline heavily on my [chess project](/en/projects/chess-engine), a multi-module Java platform I originally built in my third semester at university. With devline, I migrated the entire UI from Java Swing to JavaFX, built an animated cinematic main menu, and overhauled the server from a proof-of-concept into a production system with authentication, game persistence, and concurrent session management. The pipeline handled spec engineering, parallel TDD implementation, and review cycles for all of it.

devline is also how I'm testing whether structured AI workflows can work in a professional team setting. I built it with the goal of bringing this approach to my team at [OTTO](https://www.otto.de).

## Installation

devline is available through the Claude Code plugin marketplace.

**Step 1 — Add the marketplace.** Inside Claude Code, run:

```
/plugin marketplace add Conava/claude-devline
```

**Step 2 — Install the plugin.**

```
/plugin install devline@conava-claude-devline
```

Or open the plugin manager with `/plugin`, go to the **Discover** tab, and install from there.

**Step 3 — Verify.** Type `/devline` in any conversation. If the command is recognized, you're set.
